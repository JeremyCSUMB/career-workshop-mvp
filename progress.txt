## Codebase Patterns
- Use `$app/environment` `browser` guard before accessing `localStorage` in Svelte components
- InterviewScreen phases are component-local `$state`, not in the interview store — persist them separately to `ws_*` localStorage keys
- Svelte 5 runes: use `$state()` for reactive local state, `$derived()` for computed, `$effect()` for side effects
- Typecheck with `npx svelte-check` (no tsconfig flag needed)
- Interview store in `src/lib/stores/interview.js` auto-persists on `.update()` calls but only for store-level keys
- The template conditionally renders phases with `{#if phase === 'notes'}` etc., so restoring `phase` directly skips earlier steps
- Session lifecycle handlers (`handleLeave`, `handleSwitchSession`, `handleChangeRoom`) live in `+page.svelte` — new `ws_*` keys must be added to `clearEphemeralKeys()` and `reset()` in `interview.js`
- There are 4 `data.ended` detection paths in `+page.svelte` — ephemeral cleanup must cover all of them
- `workshop-room` GET returns the full room blob — `submissions`, `aiFollowUps`, `capabilityProfiles` are all available at `data.room.*`
- `makeEmptyRoom` in `workshop-session.js` initializes the room shape — add new fields there for new rooms
- Old rooms may lack `capabilityProfiles` (plural) — `workshop-room.js` normalizes this on read
- Resume path must call `workshop-join` before `workshop-room` — join re-registers the student, room fetches full data (submissions, AI data)
- `aiFollowUps` is array of `{ questions, timestamp }`, `capabilityProfiles` is array of `{ summary, capabilities, studentName, round }` — match by round for profile, use last entry for followups
- Pass transient server data via props to child components rather than persisting to the interview store/localStorage
- Submissions use round labels like `round1-notes`, `round1-followup` — match with `round${currentRound}-notes` pattern; the `notes` field holds the text
- SvelteKit server load functions: use `cookies.get('session')` for cookie access, `throw redirect(302, url)` for redirects
- `jsonwebtoken` works as ES module import in SvelteKit server files — SvelteKit handles CJS interop
- User data from +page.server.js load is available via `let { data } = $props()` in Svelte 5 pages
- Shared cookie parser lives in `netlify/functions/lib/cookies.js` — use `parseCookies(event.headers.cookie)` in any Netlify function
- Room blobs have optional `studentEmails` field — old rooms may lack it, always initialize with `if (!room.studentEmails) room.studentEmails = {}`
- User profile blobs stored at `user:{email}` key — contain name, email, picture, authenticatedAt
- The auth-logout endpoint already exists at netlify/functions/auth-logout.js — POST clears session cookie with Max-Age=0
- Header CSS classes: `.ws-header` (centered, max-width 640px) for interview, `.ws-header--wide` (max-width 1080px) for dashboard
- Existing CSS design system: `.ws-btn`, `.ws-btn--secondary`, `.ws-btn--small`, `.ws-btn--danger`, `.ws-card`, `.ws-link-btn`
- Theme CSS vars: `--ci-accent`, `--ci-text`, `--ci-surface`, `--ci-border`, `--ci-focus-ring`
- UserProfileMenu component: uses `onLogout` callback prop (not `createEventDispatcher`), accepts `name`/`email`/`picture`/`onLogout` props
- Svelte 5 pattern: use callback props (`onLogout`) not `createEventDispatcher` — matches existing codebase convention
- Position profile menu in headers with `.ws-header__profile` class: `position:absolute; right:24px; top:50%; transform:translateY(-50%)`
