## Codebase Patterns
- Use `$app/environment` `browser` guard before accessing `localStorage` in Svelte components
- InterviewScreen phases are component-local `$state`, not in the interview store — persist them separately to `ws_*` localStorage keys
- Svelte 5 runes: use `$state()` for reactive local state, `$derived()` for computed, `$effect()` for side effects
- Typecheck with `npx svelte-check` (no tsconfig flag needed)
- Interview store in `src/lib/stores/interview.js` auto-persists on `.update()` calls but only for store-level keys
- The template conditionally renders phases with `{#if phase === 'notes'}` etc., so restoring `phase` directly skips earlier steps
- Session lifecycle handlers (`handleLeave`, `handleSwitchSession`, `handleChangeRoom`) live in `+page.svelte` — new `ws_*` keys must be added to `clearEphemeralKeys()` and `reset()` in `interview.js`
- There are 4 `data.ended` detection paths in `+page.svelte` — ephemeral cleanup must cover all of them
- `workshop-room` GET returns the full room blob — `submissions`, `aiFollowUps`, `capabilityProfiles` are all available at `data.room.*`
- `makeEmptyRoom` in `workshop-session.js` initializes the room shape — add new fields there for new rooms
- Old rooms may lack `capabilityProfiles` (plural) — `workshop-room.js` normalizes this on read
- Resume path must call `workshop-join` before `workshop-room` — join re-registers the student, room fetches full data (submissions, AI data)
